# Регрессия промптов и смена модели

## Промпт - это код

К промпту применимы те же правила, что к любому коду в проде: версионирование, тесты, ревью, откат. Разница только в том, что промпт **выглядит** как текст, и поэтому его правят без всего перечисленного - прямо в проде, в пятницу вечером.

Типичная сцена. Клиент жалуется: агент не уточняет город доставки. Инженер дописывает в промпт строку «всегда уточняй город». Проверяет на жалобе клиента - работает. Выкатывает.

Через неделю выясняется, что агент теперь **уточняет город даже тогда, когда клиент его назвал**, и раздражает всех подряд. Правка чинила один сценарий и сломала пять, о которых никто не подумал.

Это не небрежность инженера. Это **отсутствие регрессии**. Человек физически не может удержать в голове все сценарии, на которые влияет строка в промпте.

## Как это работает

```bash
# 1. Зафиксировали текущее состояние
python -m src.harness --cases cases/support_agent.yaml --save-baseline out/baseline.json

# 2. Правим промпт

# 3. Гоняем и смотрим дельту
python -m src.harness --cases cases/support_agent.yaml --baseline out/baseline.json
```

Отчёт покажет две вещи:

**Дельту по каждой метрике** - стало лучше или хуже в целом.

**Список сломанных кейсов** - что конкретно перестало работать. Это важнее дельты: средние цифры могут не сдвинуться, а критичные сценарии - упасть.

Прогон с регрессиями **возвращает код 1**. Это и делает его пригодным для CI: правка промпта, ломающая рабочий сценарий, не доедет до прода, даже если средняя метрика выглядит прилично.

```yaml
# .github/workflows/evals.yml
- run: python -m src.harness --cases cases/support_agent.yaml --baseline evals/baseline.json
```

Промпт проходит через pull request. Как и всё остальное, что работает в проде.

## Смена модели

Вендор выпускает новую версию и обещает, что она умнее. Это может быть правдой в среднем по бенчмаркам и ложью на ваших данных.

```bash
python -m src.harness --cases cases/support.yaml --model gpt-4o --save-baseline out/gpt4o.json
python -m src.harness --cases cases/support.yaml --model claude-sonnet --baseline out/gpt4o.json
```

Одни кейсы, разные модели, одна таблица. Решение принимается по цифрам, а не по пресс-релизу.

Типичный результат такого прогона выглядит так:

| Метрика | Модель А | Модель Б | Дельта |
|---|---|---|---|
| Верных ответов | 91% | 93% | +2% |
| Опора на источники | 97% | 88% | **-9%** |
| Стоимость обращения | $0.012 | $0.004 | -$0.008 |

Модель Б **дешевле втрое и отвечает верно чаще**. И при этом она **хуже**: она чаще выдумывает. Просто её выдумки пока попадают в цель.

Без метрики groundedness этот переезд выглядел бы как чистая победа - лучше и дешевле. И через месяц агент начал бы врать клиентам, а причину искали бы где угодно, только не в «удачном» переезде.

## Дрейф моделей

Отдельная причина держать харнесс, а не прогнать его один раз.

Модели за API **меняются молча**. Вендор обновляет версию под тем же именем, подкручивает системные настройки, меняет поведение отказов. Ваш промпт не менялся, ваша база не менялась, а качество поехало.

Это не паранойя, это техническая реальность работы с чужой инфраструктурой. Единственная защита - **регулярный прогон по расписанию** и алерт при просадке.

Отсюда же вытекает довод в пользу подписки на обслуживание для клиентских внедрений: агент, установленный и забытый, деградирует не потому, что его плохо сделали, а потому, что земля под ним движется. Клиенту это честно объясняется цифрами, а не пугалками.
